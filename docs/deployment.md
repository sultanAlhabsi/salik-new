# Free private-pilot deployment

SALIK runs as one free Render web service backed by the existing Supabase
project for PostgreSQL, Auth, and private Storage. The browser and API share
the same Render origin.

## Free-tier expectations

- Render sleeps after 15 minutes without inbound traffic. The first request
  after sleep can take about one minute.
- Render's filesystem is ephemeral. Never place SQLite files, uploads, or
  other persistent state on it.
- Supabase free projects can pause after one week without activity.
- The free Supabase quotas include a 500 MB database and 1 GB of Storage.
- The free plan has no automatic database backups or production uptime SLA.

## 1. Prepare Supabase PostgreSQL

Open the Supabase project, then use **SQL Editor** to create a dedicated Prisma
role as described by the official Supabase Prisma guide. Generate a strong,
unique password in a password manager and substitute it for
`<generated-prisma-password>` only inside the SQL Editor:

```sql
create user "prisma"
  with password '<generated-prisma-password>'
  bypassrls
  createdb;

grant "prisma" to "postgres";
```

Do not reuse the database administrator password and do not save the executed
SQL with the password in this repository.

In **Connect**, select the Supavisor **Session pooler** connection string. It
must end in port `5432` and use the `prisma.<project-ref>` user. Render is a
long-running IPv4 service, so transaction mode on port `6543` is not used.

Apply these reviewed Storage migrations in timestamp order through SQL Editor:

```text
supabase/migrations/20260720084820_integrate_salik_auth_storage.sql
supabase/migrations/20260720091524_tighten_salik_storage_tenant_policy.sql
```

Confirm that the private bucket `salik-private` exists and is not public.

## 2. Create the Render service

The committed Blueprint defines one free Node.js 22 service in Frankfurt.
Open:

<https://dashboard.render.com/blueprint/new?repo=https://github.com/sultanAlhabsi/salik-new>

Authorize the `sultanAlhabsi/salik-new` GitHub repository if prompted. Keep the
service name, free plan, Frankfurt region, build command, start command, and
health path from `render.yaml`.

Render asks for these four values because they are marked `sync: false`:

| Name | Source |
| --- | --- |
| `DATABASE_URL` | Supavisor session URL using the dedicated Prisma role |
| `SUPABASE_URL` | Supabase project URL |
| `SUPABASE_PUBLISHABLE_KEY` | Supabase publishable key |
| `SUPABASE_SECRET_KEY` | Supabase server-only secret key |

Paste them only into Render's secret form. `SESSION_SECRET` and
`PAYMENT_WEBHOOK_SECRET` are generated by Render. `APP_ORIGIN` is intentionally
absent because SALIK reads Render's built-in `RENDER_EXTERNAL_URL`.

Apply the Blueprint. A successful deploy performs:

```bash
npm ci
npm run build
npm run db:migrate:deploy
npm start
```

The service becomes healthy only after PostgreSQL responds to `/api/health`.

## 3. Configure Supabase Auth redirects

After Render displays the service URL, open Supabase **Authentication → URL
Configuration**:

1. Set **Site URL** to the Render HTTPS origin.
2. Add `<render-origin>/?passwordRecovery=1` to the allowed redirect URLs.
3. Do not add localhost to production-only redirect configuration unless it is
   still required for a separate development workflow.

## 4. Bootstrap the private administrator

Create an ignored `.env.production.local` on the trusted workstation. Use the
actual values from Supabase, Render, and the password manager:

```dotenv
NODE_ENV=production
DATABASE_URL=<supavisor-session-url-on-port-5432>
APP_ORIGIN=<render-https-origin>
SESSION_SECRET=<private-local-command-secret-at-least-32-characters>
PAYMENT_WEBHOOK_SECRET=<private-local-command-secret-at-least-16-characters>
SUPABASE_URL=<supabase-project-url>
SUPABASE_PUBLISHABLE_KEY=<supabase-publishable-key>
SUPABASE_SECRET_KEY=<supabase-server-secret-key>
SUPABASE_STORAGE_BUCKET=salik-private
SALIK_SUPABASE_DISABLED=false
BOOTSTRAP_ADMIN_EMAIL=<private-owner-email>
BOOTSTRAP_ADMIN_NAME=<private-owner-name>
BOOTSTRAP_ADMIN_PASSWORD=<private-password-at-least-14-characters>
```

Load it without printing values, apply migrations, and bootstrap once:

```bash
set -a
. ./.env.production.local
set +a
npm run db:migrate:deploy
npx prisma migrate status
BOOTSTRAP_CONFIRM=SALIK_PRIVATE_PILOT npm run db:bootstrap:private
```

Expected bootstrap output is only one of:

```text
Private administrator ready: created
Private administrator ready: reconciled
```

The command is idempotent. It never clears business data and refuses to run
without production configuration, hosted Supabase Auth, and the exact
confirmation phrase. Do not run `npm run db:seed:demo` against Supabase.

## 5. Verify the live pilot

Keep the Render origin in a shell variable and verify health without exposing
credentials:

```bash
export SALIK_LIVE_ORIGIN='https://<render-service>.onrender.com'
curl --fail --silent --show-error "$SALIK_LIVE_ORIGIN/api/health"
```

Expected response:

```json
{"ok":true,"service":"salik","timezone":"Asia/Muscat"}
```

Then verify through a private browser session:

1. Sign in with the private administrator.
2. Refresh the page, sign out, and sign in again.
3. Create and read one tenant-scoped record.
4. Upload and download one private attachment.
5. Confirm an unauthorized cross-tenant request is denied.
6. Trigger a manual Render redeploy and wait for health.
7. Confirm the record and attachment still exist.
8. Inspect Render logs and browser assets for connection strings, keys,
   cookies, tokens, or passwords; none may appear.

## 6. Backup and recovery

Before important schema or data changes, export PostgreSQL from the trusted
workstation:

```bash
pg_dump "$DATABASE_URL" --format=custom --no-owner --no-acl --file="salik-$(date +%Y%m%d-%H%M%S).dump"
```

Store the dump outside the repository in encrypted storage. Free Supabase does
not provide automatic backups.

For an application regression, use Render's rollback to one of the previous
successful deploys. Database migrations are forward-only: restore a verified
export or apply a reviewed corrective migration instead of editing migration
history. If Supabase pauses for inactivity, resume it in the Dashboard and
allow Render health checks to recover.

## 7. Routine operation

- Check Render and Supabase free-tier usage monthly.
- Export the database before releases that alter schema or critical data.
- Invite team members with private, unique credentials through authenticated
  administration; never publish the demo accounts from local documentation.
- Keep the Render service, Supabase project, and GitHub repository access
  limited to the pilot operators.
